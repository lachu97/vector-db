---
title: RAG Query
api: POST /v1/query
description: Query a collection using natural language. Returns the most relevant text chunks with similarity scores.
---

<ParamField header="x-api-key" type="string" required>
  API key with any role (`admin`, `readwrite`, or `readonly`).
</ParamField>

<ParamField body="query" type="string" required>
  Natural language query text. The backend embeds this and performs a semantic search against the collection.
</ParamField>

<ParamField body="collection_name" type="string" required>
  Collection to search.
</ParamField>

<ParamField body="top_k" type="integer" default="5">
  Number of results to return.
</ParamField>

<ParamField body="filters" type="object">
  Optional metadata filter. Only return chunks whose metadata contains the specified key-value pairs.
</ParamField>

<ParamField body="include_timing" type="boolean" default="false">
  When `true`, the response includes a `timing_ms` object with `embedding_ms`, `search_ms`, and `total_ms` breakdowns.
</ParamField>

<RequestExample>
```bash cURL
curl -X POST http://localhost:8000/v1/query \
  -H "x-api-key: your-key" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "How does vector indexing work?",
    "collection_name": "articles",
    "top_k": 5,
    "filters": {"source": "docs"}
  }'
```
</RequestExample>

<ResponseExample>

```json Success
      },

        "metadata": {"source": "docs", "page": 5},

```

</ResponseExample>

<Note>
  Unlike the `/v1/collections/{name}/search` endpoint which requires a raw vector, the RAG query endpoint accepts plain text. The backend handles embedding the query internally before performing the search.
</Note>
